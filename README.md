# CadExchange Ticket Reservation (Simplified)

Упрощенный API-сервис бронирования мест с высокой конкурентностью.

## Что реализовано

- `POST /reserve` принимает `user_id` и `seat_id`.
- Логика бронирования: одно место (`seat_id`) может быть забронировано только один раз.
- Гонка (Race Condition) исключена атомарным Lua-скриптом в Redis (`SET if not exists` в рамках одного вызова).

## Почему выбран Redis, а не PostgreSQL

При старте продаж до 50 000 одновременных пользователей фокус на “горячей” строке/ключе (один и тот же `seat_id`).
Redis позволяет:

- атомарно обработать конкуренцию в одном вызове (Lua под капотом выполняется последовательно внутри Redis),
- избежать тяжёлых блокировок строк в БД (как при `SELECT ... FOR UPDATE` в Postgres),
- выдерживать большой поток коротких запросов.

Если нужно хранить бронь как источник правды на длительный срок, в Redis можно включить с AOF/RDB (в compose используется AOF).

## Запуск

1. Поднимите Redis:
   - `docker compose up`
2. Установите зависимости:
   - `npm i`
3. Запустите сервис:
   - `cp .env.example .env`
   - `npm run dev`

По умолчанию: `http://localhost:3000`.

## Пример запроса

```bash
curl -s -X POST http://localhost:3000/reserve \
  -H "content-type: application/json" \
  -d '{"user_id": 123, "seat_id": 42}'
```

## Нагрузочный тест (опционально)

Пакет `autocannon`:

```bash
SEAT_ID=42 CONNECTIONS=2000 DURATION_SEC=20 npm run loadtest
```

Тест специально пытается зарезервировать один и тот же `seat_id`, чтобы проверить отсутствие "двойной брони".
